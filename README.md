## 1. プロジェクトの背景と目的  
**【背景】**  
銀行口座を持たない人々に対し、安心で快適な融資体験を提供することで、金融包摂の拡大を目指しています。こうした金融サービスが行き届いていない人々が満足のいく融資体験を得られるよう、Home Creditは通信事業者情報や取引情報など、さまざまな代替データを活用して**顧客の返済能力を予測**しています。

<br>

* **データ提供元**:Home Credit Group（銀行口座を持たない人や信用履歴が少ない層に対し、POSローン（店頭分割払い）やアプリを用いた簡易ローンを提供）
* **主催**: Home Credit Group
<br>

**【目的】**  
本プロジェクトでは、ローン審査における顧客の返済能力を予測し、企業方針(信用情報が薄い層へのローン提供)を維持しつつ、利益改善を目的とする。
データの制約(融資申請却下された人のデータ欠如)から全自動判定は当該コンペにおいて現実的でないため、現状の審査に追加スクリーニングとしてモデル判定を実施。
債務不履行のリスクが低い層を追加承認し、リスクが高い層は人間による2次審査に移行以降することで利益改善と企業方針を両立させる。
<br>

## 2.ビジネスプラン：審査最適化の設計  
<pre>
【業務処理フロー】
1次審査通過者(現状の審査)  
　　│  
　　▼  
  モデルスコアリング（予測確率算出）  
  　│  
　　├─ 予測確率 < 0.23 ──→ 自動承認（全体の約85%）  
　　│  
　　└─ 予測確率 ≥ 0.23 ──→ 2次審査（人間による審査、約15%）  
　　　　　　　　　　　　　　　　　│  
　　　　　　　　　　　　　　　　　├─ 承認  
　　　　　　　　　　　　　　　　　└─ 却下  
</pre>
正常Precisionを高めるとデフォルトPrecisionが低下し（閾値0.24時点で27.3%）、モデル単独での自動却下は非現実的。  
また、正常Precisionの要求を高めるほど自動承認件数が減少し、モデル導入の効果が薄れる。  
なお、2次審査をなくし、モデル単独で自動却下を行った場合でも利益の改善率は22%程度見込めるが、デフォルト判定されたうち約7割の顧客は返済可能にも関わらず、融資を却下することとなり、企業理念に反すると判断。  
これらの観点から人間による2次審査との組み合わせが前提となる。  
※補足  
・起点を「1次審査通過者」とする理由：本データは融資実行済みの顧客のみで構成されており、審査段階で却下された申請を含まないため、モデルの適用範囲は1次審査を通過した申請者に限定される。(データ上の限界)  
・閾値 0.23 の選定根拠  
閾値0.23は利益改善率のピーク付近に位置し、前後の閾値変動に対しても改善率が安定しているロバストなポイントとして選定した。

■ 純改善額の算出
  1. ベースライン（モデルなし：全件承認）
     = 全正常顧客の融資額 × 利益率 − 全デフォルトの融資額 × LGD

  2. モデル導入時
     = TN（正しく承認した正常顧客）の融資額 × 利益率
       − FN（見逃しデフォルト）の融資額 × LGD
       − 審査コスト

  3. 純改善額 = モデル導入時 − ベースライン
  
■ 前提条件  
業務処理フロー上でデフォルト判定(TP,FP)したものは  
人間による2次審査に移行するため計算式からは除外。  
  - TP（検知デフォルト）は人間審査で100%却下される想定  
  　実際には人間審査でも見逃しが発生するため、上記の純改善額は理論上の上限値となる。  
  - FP（誤検知した正常顧客）の利益は全額損失として計上（保守的）
    実際には審査通過後に承認されるケースがあるため、機会損失は過大に見積もられている。  
この2つは逆方向に相殺しあうため、影響はある程度限定的になると思料されるが、相殺の程度は人間審査精度に依存するので、運用開始後に検証を要す。

## 5.主な発見と学び

1.EDA・FE  
　-寄与度の高い特徴量同士は情報が重複し、合成しても精度向上に繋がらない場合がある。また細分化はサンプル不足による過学習を招くため、Coverage（被覆率）の確認が必要。  
2.技法・検討方法  
　-ハイパーパラメータチューニング(Optuna)、SHAP、Target Encoding、リーク対策(Out-of-Fold＋Smoothing)  
3.試験的デプロイ導入  
　-AWSでのデプロイ及びGITHUBを用いたCI/CDの概要及び手順。また、zipデプロイの250MB制限。

## 6. 制約と今後の課題
【データ上の制約】  
・本データは融資実行済みの顧客のみで構成されており、審査段階で却下された申請は含まれない（サバイバーシップバイアス）。1次審査の自動化を行うためには却下データを含めた再学習・構築修正が必要  
・通貨単位が不明のため、コスト試算は相対比較での提示に留まる。パラメータを実データに置き換えた際の再試算が必須  
・単一時点のデータであり、時系列的な変動（景気変動、季節性）を捉えていない。本番運用ではモデルドリフトの監視が必要  
【モデル上のリスク】  
・SHAP寄与度上位をEXT_SOURCE由来の特徴量が占めており、外部スコアの提供元が変更・停止した場合の影響が大きい  
・ORGANIZATION_TYPE（業種）はSHAP分析でばらつきが大きく、サンプル数の偏りによる過学習リスクがある  
【今後の改善余地】  
・サブテーブル（POS/installments/credit_card）の集約が一律min/max/mean/sum/stdに留まっており、ここの深堀で精度向上が見込める
・デプロイはLGBM Fold0単体での推論であり、本番ではアンサンブル構成の検討とレイテンシ・メモリのバランス調整が必要  
・API認証（APIキー/IAM）、入力バリデーション、モデルバージョニングが未実装

## 7. デプロイ構成、実行方法  
【構成】  
<pre>
 GitHub → GitHub Actions → ECR → Lambda ← S3(モデル)  
 　　　　　　　　　　　　　　　　　　　↓  
 　　　　　　　　　　　　　　　　API Gateway → レスポンス(JSON)  

【実行手順】  
  curl -X POST \  
    https://{api-id}.execute-api.ap-northeast-1.amazonaws.com/prod/predict \  
    -H "Content-Type: application/json" \  
    -d '{"features": {"EXT_SOURCE_2": 0.5, "EXT_SOURCE_3": 0.3}}'  

  レスポンス例:  
    {  
      "probability": 0.073,  
      "threshold": 0.24,  
      "decision": "auto_approve",  
      "features_received": 2  
    }  
</pre>

※APIの制約事項  
・現在のAPIはFE済みデータ（669特徴量）を入力とするデモ版であり、生の申請データからの推論には対応していない  
・テストコードではカテゴリ型特徴量が除外されており（669→631）、本来の予測精度より低下している  
・実運用では前処理パイプライン（特徴量生成・Target Encoding等）をLambda内またはStep Functionsで構築する必要がある  
・デプロイはLGBM Fold0単体での推論であり、アンサンブル構成（LGBM+CatBoost）の組み込みは未実装
